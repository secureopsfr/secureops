Vérification DNS du domaine

Pour les modes de scan non passifs (intrusif, personnalisé, destructeur, etc.), SecureOps peut exiger une preuve que vous contrôlez la zone DNS du site cible. Cette preuve se fait en publiant un enregistrement TXT unique fourni par l’application — comme pour d’autres services (certificats, Search Console).

Cette page résume le fonctionnement côté utilisateur. La spécification technique complète (API, base de données, variables d’environnement) est dans le dépôt : docs/VERIFICATION-AUTORISATION.md.

1. Pourquoi cette vérification ?

Les scans actifs envoient des requêtes qui peuvent être assimilées à des tests d’intrusion. Limiter ces scans aux personnes capables de publier un enregistrement DNS sur le domaine cible réduit le risque d’abus sur des sites dont l’utilisateur n’est pas légitime.

Autorisation d’audit

Cette vérification DNS ne remplace pas une autorisation contractuelle ou légale d’audit : ne testez que des applications que vous êtes autorisé à évaluer.

2. Quand est-elle exigée ?

  • Scan passif : la vérification DNS n’est pas demandée.
  • Tous les autres modes (intrusif, personnalisé, destructeur, etc.) : elle est obligatoire — le scan ne peut pas être lancé tant que le domaine dérivé de l’URL n’est pas vérifié pour votre compte.

3. Étapes dans le hub Scanner

  1. Saisissez l’URL à scanner pour un scanner autre que le scanner passif.
  2. Dans la section Vérification du domaine (DNS), cliquez sur Générer les instructions DNS. Un secret et le nom d’hôte TXT à créer s’affichent.
  3. Chez votre fournisseur DNS (registrar, Cloudflare, etc.), créez un enregistrement TXT avec exactement le nom et la valeur indiqués.
  4. Attendez la propagation (souvent quelques minutes ; parfois plus).
  5. Revenez sur SecureOps et cliquez sur Vérifier le TXT. Le service interroge le DNS public : si une chaîne correspond au secret attendu, le domaine est enregistré pour votre compte.

Vous pouvez réessayer la vérification sans régénérer le secret tant que la demande (challenge) n’a pas expiré.

4. Enregistrement TXT attendu

  • Type : TXT
  • Nom / hôte : _secureops-verify comme sous-domaine du domaine à prouver (ex. _secureops-verify.example.com)
  • Valeur : le jeton affiché par SecureOps (ex. préfixe secureops-verify- suivi de caractères aléatoires)

Plusieurs enregistrements TXT sur le même nom sont possibles : la vérification réussit si au moins une valeur correspond au jeton attendu.

5. Quel « domaine » est utilisé ?

À partir de l’URL, le système calcule le domaine enregistrable (souvent l’eTLD+1, par ex. example.com pour www.example.com). C’est ce domaine qui doit être vérifié et qui apparaît dans la liste des domaines validés pour votre compte.

Si votre politique impose de prouver un hôte précis (ex. app.client.com), le nom TXT sera _secureops-verify.app.client.com — suivez les instructions affichées après génération.

6. Après la vérification

  • La vérification est enregistrée avec une date d’expiration (renouvellement périodique possible selon la configuration).
  • Un même domaine ne peut être associé qu’à un seul compte à la fois au sein de la plateforme.
  • Les scans planifiés non passifs peuvent également être ignorés ou refusés si le domaine n’est pas vérifié, selon la même règle.

7. Limites et bonnes pratiques

  • Les résolveurs DNS publics peuvent mettre du temps à voir votre TXT : en cas d’échec, vérifiez la zone, puis réessayez.
  • Ne partagez pas le jeton hors de votre équipe autorisée à gérer le DNS.
  • Pour l’intégration technique (assert interne, scan-service, tables Postgres), voir la documentation développeur dans le dépôt.

Intrusive Scan

The intrusive scan goes beyond passive observation: it actively tests your defences by simulating real attacks — injections, authentication bypasses, API abuse, business logic manipulation. This mode is designed for teams that want to know whether their protections hold up, not just whether they are configured. Run only on applications you are authorised to test.

1. Principle of the intrusive scan

The intrusive scan sends deliberately malformed or malicious requests to observe how your application responds. Unlike the passive scan which reads, the intrusive scan interacts: it injects payloads, manipulates parameters, and forces unintended behaviours.

What the intrusive scan detects

  • Injection flaws (SQL, NoSQL, OS command, SSTI, XXE, LFI/RFI, path traversal)
  • Access control issues (dangerous CORS, CSRF, SSRF, Host Header injection, open redirects)
  • Authentication flaws (IDOR, brute force without lockout, sessions not invalidated, JWT without expiry)
  • Client-side vulnerabilities (reflected XSS via parameters)
  • Business logic flaws (race conditions, missing rate limiting, upload abuse, mass assignment)
  • Modern API weaknesses (GraphQL introspection, gRPC Reflection, WebSocket without auth)
  • Infrastructure issues (cache poisoning, request smuggling, exposed internal endpoints, public object storage)

What the intrusive scan does NOT do

  • It does not delete, modify or create data in production (payloads are built to observe, not persist)
  • It does not test low-level network flaws (port scanning, system exploits)
  • It does not break cryptographic protections
  • It does not perform out-of-band attacks (DNS callbacks, out-of-band exfiltration) — these are planned for a future version

Active scan = traces in logs

An intrusive scan leaves traces in your application and server logs (requests with SQL, XSS payloads, etc.). This is normal and expected. If you have security alerts or a WAF, notify your team before launching the scan to avoid false alarms.

2. Prerequisites and authorisations

The intrusive scan is a professional security testing tool. Its use is governed by strict rules:

  • Written authorisation required: you must own the application or hold explicit authorisation from the owner. Testing an application without authorisation is illegal in most jurisdictions.
  • Recommended environment: prefer running the scan on a staging or pre-production environment. In production, payloads may trigger alerts, fill logs or interact with real workflows.
  • Scan report: the generated PDF report contains proof of detected vulnerabilities. Keep it securely — it documents your application's security posture at a given point in time.

Authorised use only

SecureOps cannot be used to test third-party applications without authorisation. Any malicious or unauthorised use is the sole responsibility of the user.

DNS domain verification (active modes)

In production, non-passive scans may require proof of DNS control for the target domain: you publish a TXT record provided by SecureOps. For a step-by-step walkthrough (Scanner hub, propagation, good practices), see the DNS verification guide.

3. Using target credentials (optional)

By default, the intrusive scan runs unauthenticated: it tests what an external attacker can see and exploit. To go further and test the protected parts of your application, you can provide credentials.

Supported credential types

  • Cookie string: the full value of the Cookie header from an active session. For example: session=abc123; csrf_token=xyz789. Obtain it from browser DevTools (Network tab, headers of any authenticated request).
  • Bearer token: a JWT or OAuth token to include in the Authorization: Bearer <token> header. Useful for REST APIs with token-based authentication.

What credentials change

  • Scan requests will automatically include the provided credentials
  • Protected pages and endpoints will be accessible, enabling post-login access control testing (IDOR, BOLA, BFLA)
  • Session checks (invalidation after logout, JWT expiry) will be more relevant

Credentials are not persisted

Provided credentials are used only during the scan execution and are never stored. Use a dedicated test account with limited rights — never provide production administrator credentials.

4. Frontend vs backend target

The intrusive scan adapts the checks it runs based on the selected target type:

Target Included checks Excluded checks
Frontend Reflected XSS, CORS, CSRF, open redirects, SSRF, HTTP methods, injections, sessions, upload, business logic, infra GraphQL, gRPC, WebSocket, OpenAPI (backend protocols)
Backend (API) All checks including GraphQL, gRPC, WebSocket, OpenAPI Reflected parameters (XSS) specific to HTML

In practice: if your target URL is a regular website (yoursite.com), choose Frontend. If it is a REST/GraphQL/gRPC API (api.yoursite.com), choose Backend.

5. Injections

Injections are the most critical family of vulnerabilities. An attacker makes your application or its database interpret data as code.

SQL Injection (SQLi)

The scan injects SQL payloads into GET, POST parameters and headers (User-Agent, Referer, X-Forwarded-For). It detects:

  • Database errors in the response (MySQL, PostgreSQL, MSSQL, OracleDB, SQLite messages)
  • Time-based injections: if a SLEEP(1) / pg_sleep(1) payload causes an abnormal response delay, the vulnerability is confirmed

Severity: Critical. A SQLi allows reading, modifying or deleting the entire database, or even executing system commands.

NoSQL Injection (NoSQLi)

For JSON APIs, the scan tests MongoDB operators ($gt, $where, $regex) in parameters and request bodies to bypass query filters.

Severity: High to critical depending on the achievable bypass.

OS Command Injection

Command separators (;, &&, |, backticks) combined with Unix commands (id, sleep, whoami) are injected into parameters. Detection is based on responses and delays (time-based for non-reflected commands).

Severity: Critical. Arbitrary code execution on the server.

Server-Side Template Injection (SSTI)

Template expressions ({{7*7}}, #{7*7}, ${7*7}) are injected into parameters. If the response contains 49 or a template error, the rendering engine is vulnerable (Jinja2, Twig, FreeMarker, etc.).

Severity: Critical. SSTI leads directly to server-side code execution.

XXE (XML External Entity)

For endpoints that accept XML, the scan sends an external entity (&xxe;) defined in the DOCTYPE. If the response contains the entity's content or produces a characteristic error, the XML parser is vulnerable.

Severity: Critical. Allows reading local server files, triggering SSRF or causing denial of service (billion laughs).

LFI / RFI (Local / Remote File Inclusion)

Directory traversal sequences and system file paths (/etc/passwd, C:\Windows\win.ini) are tested in parameters that appear to reference files. Detection looks for system file signatures in responses.

Severity: Critical (LFI can read server secrets) to high (RFI can include remote code).

Path Traversal

Sequences of ../ encoded in multiple ways are injected into path parameters to attempt access to files outside the web root.

Severity: High. Reading configuration files, logs or secrets outside the allowed directory.

What can skew results

An active WAF may block payloads before they reach your application — injections will not appear in results even if the code is vulnerable. A WAF that hides SQL errors without fixing the underlying code can also mask a real vulnerability.

6. Access Control

Active CORS (reflection + credentials)

The scan sends requests with a controlled Origin header to your application and checks the response:

  • Origin reflection + credentials: if the application responds with Access-Control-Allow-Origin: [your-origin] and Access-Control-Allow-Credentials: true, a malicious site can make authenticated requests on behalf of your users and read the responses.
  • Wildcard on sensitive endpoint: Access-Control-Allow-Origin: * on an API route exposes response data to any site.

Severity: High to critical depending on the exposed data.

CSRF (Cross-Site Request Forgery)

The scan analyses POST forms on sensitive endpoints (login, password change, payment) looking for:

  • Absence of a CSRF token or anti-CSRF header
  • Absence of the SameSite attribute on session cookies (which provides partial protection)

An endpoint vulnerable to CSRF can be triggered without the user's knowledge from any site.

Severity: Medium to high depending on the achievable actions.

SSRF (Server-Side Request Forgery)

URLs pointing to internal addresses (http://169.254.169.254/ — AWS metadata, http://localhost/, http://192.168.x.x/) are injected into parameters that seem to contain URLs. If the response reveals internal content or a characteristic connection error, the check is positive.

Severity: Critical in cloud environments. SSRF allows reaching internal services, cloud metadata (IAM keys, tokens) and non-publicly exposed services.

Host Header Injection

The scan sends requests with a forged Host header. If the response reflects the Host value or generates URLs based on it, the application may be vulnerable to password-reset poisoning attacks (sending reset links pointing to an attacker's domain).

Severity: Medium to high depending on usage.

Open Redirects

Redirect parameters (redirect, next, url, return, etc.) are tested with controlled external URLs. If the application redirects to a third-party URL without validation, an attacker can create phishing links on your trusted domain.

Severity: Low to medium. Facilitates phishing by abusing trust in your domain.

Dangerous HTTP methods

The scan sends OPTIONS requests then tests PUT, DELETE, TRACE on detected endpoints. TRACE can be exploited to steal cookies via script injection (XST attack). Unprotected PUT/DELETE expose resources to modification or deletion.

Severity: Low to medium. Disable if not necessary.

7. Authentication & Sessions

IDOR / BOLA / BFLA

The scan attempts to access resources belonging to other users by modifying numeric identifiers (/api/users/1, /api/orders/42) in URLs. It also checks whether administration endpoints are accessible without privilege escalation.

  • IDOR (Insecure Direct Object Reference): accessing an object by its ID without rights verification
  • BOLA (Broken Object Level Authorization): API-first variant of IDOR
  • BFLA (Broken Function Level Authorization): accessing admin functions without the required rights

Severity: High to critical. These flaws represent the top vulnerability category in the OWASP API Security Top 10.

Brute Force / Account Lockout

The scan sends multiple successive login attempts with invalid credentials and checks whether the application blocks or slows down attempts after N failures. The absence of an account lockout mechanism exposes the application to dictionary attacks.

Severity: High. Without lockout, an attacker can test thousands of password/login combinations.

Session Management / JWT

The scan checks:

  • Session not invalidated: after a logout call, is the old session cookie still accepted?
  • Missing JWT expiry: JWT tokens without an exp field remain valid indefinitely
  • JWT algorithm none: acceptance of unsigned tokens (known vulnerability in misconfigured JWT libraries)
  • Session fixation: does the session identifier change after authentication?

Severity: High. A non-invalidated token allows an attacker to continue using a stolen session even after the victim has logged out.

OAuth / OIDC

The scan tests common OAuth flows looking for:

  • Absence of state parameter verification (CSRF protection for the OAuth flow)
  • Redirect to unregistered URIs (redirect_uri not validated)
  • Access tokens exposed in URLs (in logs, browser history)

Severity: High. These flaws can allow OAuth access token theft and account compromise.

8. Client-side Injections (Reflected XSS)

The scan injects XSS markers (<script>alert(1)</script>, encoded variants) into GET and POST parameters. It then analyses the HTML response to detect whether the marker is reflected without encoding.

What is tested

  • Search parameters (?q=, ?search=, ?keyword=)
  • Navigation parameters (?page=, ?sort=, ?filter=)
  • HTML form bodies

Why reflected XSS is dangerous

A reflected XSS allows an attacker to send a crafted link to a victim. When the victim clicks it, the script executes in their browser with your site's privileges: session cookie theft, redirection to a phishing site, actions executed on their behalf.

Severity: High. If session cookies without HttpOnly are present, severity rises to critical.

Limitations of the XSS check

The scan detects direct reflection in HTML responses. It does not detect stored XSS (injected into the database) or DOM-based XSS (client-side JavaScript manipulation). These variants require deeper analysis with a full browser.

9. Business Logic & Data

Business Logic

The scan probes transaction-type endpoints to detect:

  • Negative amounts/quantities accepted: sending quantity=-1 or amount=-100 in an order may create an unintended credit
  • Workflow state manipulation: direct access to a process step, skipping previous steps

Severity: High to critical. These flaws are specific to your application and often invisible to classic automated tools.

Race Conditions

The scan sends requests in parallel to critical endpoints (coupon validation, purchase, transfer) to detect inconsistent behaviours due to unsynchronised concurrent accesses. A race condition may allow spending the same coupon twice or withdrawing more than the balance.

Severity: High depending on the affected features.

DoS / Rate Limiting

The scan sends a burst of 20 rapid requests to the target URL to detect the absence of rate abuse protection. The absence of a 429 Too Many Requests response after this burst indicates that an attacker can saturate the application or bypass usage limits.

Severity: Low to medium. An unrate-limited endpoint can be abused for scraping, enumeration or DoS.

Upload Abuse

The scan tests file upload endpoints with:

  • Files whose extension and Content-Type are inconsistent (e.g. a PHP file disguised as an image)
  • Large files to test size limits
  • Paths with directory traversal in the filename

Severity: Critical if a PHP/JSP file can be uploaded and executed — this leads to arbitrary code execution.

Insecure Deserialization

The scan detects endpoints that accept serialised objects (Java, PHP, Python pickle, YAML) and sends known malformed payloads to trigger characteristic errors. Detection remains conservative (rare false positives) — it does not attempt real exploitation.

Severity: Critical. Insecure deserialization is one of the most direct paths to server-side code execution.

Mass Assignment

The scan adds extra fields to PUT/PATCH requests (role, is_admin, price, balance) to detect whether the application accepts them without filtering. A framework that directly binds the JSON body to an object's properties is vulnerable if unexpected fields are not explicitly ignored.

Severity: High. Allows privilege escalation or manipulation of sensitive data.

10. APIs & Modern Protocols

These checks target API endpoints and advanced protocols. They are enabled for Backend targets.

GraphQL — Exposed Introspection

GraphQL introspection allows listing all types, fields and mutations in your schema. The scan detects if it is enabled in production by sending a standard introspection query. An exposed schema is a complete attack guide to your API.

Severity: Medium. Disable in production.

GraphQL — Query Depth Abuse

The scan sends deeply nested GraphQL queries to test whether the application limits query depth. Without a limit, a query with 50 nesting levels can cause exponential load (DoS).

Severity: Medium.

GraphQL — Unauthenticated Subscriptions

The scan attempts to subscribe to GraphQL subscriptions without providing credentials. If real-time data is accessible anonymously, this may expose sensitive events.

Severity: High if user data is exposed.

gRPC Reflection Exposed

The gRPC Reflection service lists all services and methods available on your gRPC server. The scan detects if this service is active in production. Like GraphQL introspection, it is a complete attack guide to your gRPC API.

Severity: Medium. Disable in production.

WebSocket — Authorisation

The scan tests whether your WebSocket endpoint accepts connections without checking the Origin header or without authentication, which would allow a malicious site to establish a WebSocket connection on behalf of an authenticated user (WebSocket hijacking).

Severity: High depending on the data exchanged on the channel.

OpenAPI Schema Without Validation

The scan retrieves your exposed OpenAPI schema and tests whether documented endpoints accept data that does not conform to the schema (types, formats, constraints). Insufficient validation can allow injections or unintended behaviours.

Severity: Variable. The publicly accessible OpenAPI schema is itself an informational finding.

11. Infrastructure & Cache

Cache Poisoning

The scan injects proxy headers (X-Forwarded-Host, X-Host, X-Forwarded-Scheme) with controlled values and checks whether the response reflects them. If the cache stores a poisoned response, all users who subsequently access the same page receive the malicious version.

Severity: High. Cascading impact on all users served by the cache.

Web Cache Deception

The scan accesses URLs like /profile/fake.css to detect whether your application serves dynamic authenticated pages under static extensions (CSS, JS, PNG) that would be publicly cached. This would allow an attacker to access private pages served from the cache.

Severity: High. Exposure of user data via a public cache.

Request Smuggling (CL/TE and TE/TE)

The scan sends requests with ambiguous Content-Length and Transfer-Encoding headers to test whether the front-end proxy and back-end server disagree on request boundaries. A disagreement allows injecting data into another user's request.

Severity: Critical. Allows bypassing security protections, injecting malicious requests or stealing other users' sessions.

Exposed Internal Endpoints

The scan probes paths known to expose diagnostic information in production: /actuator, /actuator/env, /actuator/health, /_debug, /debug/vars, /metrics, /__admin. These Spring Boot, Django Debug Toolbar or other framework endpoints often reveal environment variables, internal connections or internal metrics.

Severity: High. /actuator/env may contain database passwords, API keys, configuration secrets.

Public Object Storage (S3 / GCS / Azure Blob)

The scan detects storage buckets referenced in responses and attempts to list their contents. A public bucket with listing enabled exposes all files it contains — user documents, exports, backups.

Severity: Critical if listing is active. High if the bucket exists but is properly protected.

12. Understanding your intrusive score

Severity levels

Identical to the passive scan, severity levels reflect the seriousness of an exploitable vulnerability:

Severity Penalty Description
Critical100 pointsDirect exploitation: code execution, full database access, account takeover. Fix immediately.
High50 pointsUnauthorised data access, authentication bypass, serious exposure. Priority treatment.
Medium25 pointsExploitable vulnerability under specific conditions. Plan quickly.
Low10 pointsEnlarged attack surface without obvious direct exploitation. Good practice to fix.
Info0 pointsSecurity observation, revealing configuration but not directly exploitable.

Weighting by macro-category

The most critical categories (injections, SSRF, IDOR) have a higher weight in the overall score calculation:

CategoryChecksRelative weight
Injections (SQLi, NoSQLi, commands, SSTI, XXE, LFI, Path Traversal)14 checksHigh
Authentication & Sessions (IDOR, Brute Force, JWT, OAuth)7 checksHigh
Access Control (CORS, CSRF, SSRF, Host Header, Redirects, HTTP Methods)6 checksHigh
Business Logic & Data (Race, DoS, Upload, Deserialization, Mass Assignment)7 checksMedium
Client-side Injections (Reflected XSS)1 checkMedium
Infrastructure & Cache (Cache Poisoning, Smuggling, Internal Endpoints, Storage)5 checksMedium
APIs & Modern Protocols (GraphQL, gRPC, WebSocket, OpenAPI)6 checksLow to medium

Interpreting the score

A score of 100 indicates no anomalies were detected by the intrusive scan checks. This is a good signal, not an absolute guarantee: the scan covers vulnerabilities detectable automatically with in-band requests. Complex business logic flaws or out-of-band vulnerabilities may require additional manual assessment.

Priority: always fix critical findings first, regardless of their impact on the score. A critical SQL injection is more urgent than a dozen low findings.

13. Limitations and false positives

What the scan cannot detect

  • Out-of-band vulnerabilities: payloads that exfiltrate data via DNS or HTTP callbacks are not detected (planned for a future version)
  • Stored XSS: requires a browser to observe execution in the DOM
  • Flaws in code not accessible via HTTP: vulnerabilities in asynchronous jobs, workers, non-exposed processes
  • Complex business logic: the scan can detect common patterns but does not understand the semantics specific to your domain
  • Multi-factor authentication: MFA flows cannot be simulated automatically

Managing false positives

Some findings may be false positives depending on your context:

  • Missing rate limiting: if you have rate limiting protection at CDN or load balancer level (Cloudflare, Nginx), it may not be visible at the application level and generate a false positive
  • Dangerous HTTP methods: PUT/DELETE are normal and expected on a well-designed REST API — their presence is only a problem if not protected by authentication
  • IDOR / BFLA: checks test access to common sequential IDs. On a system with UUID or opaque token IDs, checks may not cover all cases

14. What can block or impact the scan

WAF and anti-bot protections

A Web Application Firewall (Cloudflare, AWS WAF, ModSecurity) can block injection payloads before they reach your application. The scan will receive 403 responses or challenge pages instead of application responses. In this case:

  • Injections will not appear in results even if the code is vulnerable
  • This is a sign your WAF is working — but the underlying code still needs fixing
  • For a complete test, configure an IP allowlist to bypass the WAF during the scan

Timeouts

The intrusive scan has a higher time budget than the passive scan (up to 8 minutes). Time-based checks (blind SQLi, command injection) intentionally wait for abnormal delays. On a very slow application, time-based false positives may appear.

Environments without parameters

Some checks (injections, reflected XSS) need parameters in the URL or HTML forms to have something to test. On a static page without parameters, these checks will run but produce no result (neither positive nor negative).

REST-only APIs without documentation

On an API without exposed Swagger/OpenAPI, the scanner can only explore the provided URL and paths it attempts to infer. For complete API coverage, use the API documentation import feature (OpenAPI/Postman) to provide the list of endpoints to scan.

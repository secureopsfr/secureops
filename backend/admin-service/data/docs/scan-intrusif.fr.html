Scan intrusif

Le scan intrusif va plus loin que l'observation passive : il teste activement vos défenses en simulant des attaques réelles — injections, contournements d'authentification, abus d'API, manipulation de logique métier. Ce mode est conçu pour les équipes qui veulent savoir si leurs protections résistent, pas seulement si elles sont configurées. À n'exécuter que sur des applications que vous êtes autorisé à tester.

1. Principe du scan intrusif

Le scan intrusif envoie des requêtes délibérément malformées ou malveillantes pour observer comment votre application réagit. Contrairement au scan passif qui lit, le scan intrusif interagit : il injecte des payloads, manipule des paramètres, force des comportements non prévus.

Ce que le scan intrusif détecte

  • Les failles d'injection (SQL, NoSQL, commande OS, SSTI, XXE, LFI/RFI, path traversal)
  • Les problèmes de contrôle d'accès (CORS dangereux, CSRF, SSRF, Host Header injection, redirections ouvertes)
  • Les failles d'authentification (IDOR, bruteforce sans verrouillage, sessions non invalidées, JWT sans expiration)
  • Les vulnérabilités côté client (XSS réfléchi via paramètres)
  • Les failles de logique métier (race conditions, absence de rate limiting, abus d'upload, mass assignment)
  • Les faiblesses d'APIs modernes (GraphQL introspection, gRPC Reflection, WebSocket sans auth)
  • Les problèmes d'infrastructure (cache poisoning, request smuggling, endpoints internes exposés, object storage public)

Ce que le scan intrusif ne fait pas

  • Il ne supprime, ne modifie et ne crée pas de données en production (les payloads sont construits pour observer, pas pour persister)
  • Il ne teste pas de failles réseau de bas niveau (scan de ports, exploits système)
  • Il ne contourne pas les protections cryptographiques
  • Il ne réalise pas d'attaques out-of-band (callbacks DNS, exfiltration hors-bande) — ces vecteurs sont prévus pour une version ultérieure

Scan actif = traces dans les logs

Un scan intrusif laisse des traces dans vos journaux applicatifs et serveur (requêtes avec payloads SQL, XSS, etc.). C'est normal et attendu. Si vous avez des alertes de sécurité ou un WAF, prévenez votre équipe avant de lancer le scan pour éviter les fausses alertes.

2. Prérequis et autorisations

Le scan intrusif est un outil de test de sécurité professionnel. Son utilisation est encadrée :

  • Autorisation écrite obligatoire : vous devez être propriétaire de l'application ou disposer d'une autorisation explicite du propriétaire. Tester une application sans autorisation est illégal dans la plupart des juridictions.
  • Environnement recommandé : préférez lancer le scan sur un environnement de staging ou de recette. En production, les payloads peuvent déclencher des alertes, saturer des logs ou interagir avec des workflows réels.
  • Rapport de scan : le rapport PDF généré contient les preuves des vulnérabilités détectées. Conservez-le de façon sécurisée — il documente l'état de sécurité de votre application à un instant donné.

Usage autorisé uniquement

SecureOps ne peut pas être utilisé pour tester des applications tierces sans autorisation. Tout usage à des fins malveillantes ou non autorisées engage la responsabilité exclusive de l'utilisateur.

Vérification DNS du domaine (modes actifs)

En production, les scans non passifs peuvent être soumis à une preuve de contrôle DNS sur le domaine cible : vous publiez un enregistrement TXT fourni par SecureOps. Pour la procédure pas à pas (hub Scanner, délais, bonnes pratiques), consultez le guide Vérification DNS.

3. Utiliser des credentials cible (optionnel)

Par défaut, le scan intrusif s'exécute de manière non authentifiée : il teste ce qu'un attaquant externe peut voir et exploiter. Pour aller plus loin et tester les parties protégées de votre application, vous pouvez fournir des credentials.

Types de credentials supportés

  • Cookie string : la valeur complète de l'en-tête Cookie d'une session active. Par exemple : session=abc123; csrf_token=xyz789. Obtenez-le depuis les DevTools du navigateur (onglet Réseau, en-têtes de n'importe quelle requête authentifiée).
  • Bearer token : un JWT ou token OAuth à inclure dans l'en-tête Authorization: Bearer <token>. Utile pour les APIs REST avec authentification par token.

Ce que les credentials changent

  • Les requêtes du scan incluront automatiquement les credentials fournis
  • Les pages et endpoints protégés seront accessibles, permettant de tester les contrôles d'accès post-login (IDOR, BOLA, BFLA)
  • Les checks de session (invalidation après déconnexion, expiration JWT) seront plus pertinents

Credentials non persistés

Les credentials fournis sont utilisés uniquement pendant l'exécution du scan et ne sont jamais stockés. Utilisez un compte de test dédié avec des droits limités — ne fournissez jamais les credentials d'un compte administrateur de production.

4. Cible frontend vs backend

Le scan intrusif adapte les checks exécutés selon le type de cible sélectionné :

Cible Checks inclus Checks exclus
Frontend XSS réfléchi, CORS, CSRF, redirections ouvertes, SSRF, méthodes HTTP, injections, sessions, upload, logique métier, infra GraphQL, gRPC, WebSocket, OpenAPI (protocoles backend)
Backend (API) Tous les checks, y compris GraphQL, gRPC, WebSocket, OpenAPI Paramètres réfléchis (XSS) spécifiques au HTML

En pratique : si votre URL cible est un site web classique (votresite.com), choisissez Frontend. Si c'est une API REST/GraphQL/gRPC (api.votresite.com), choisissez Backend.

5. Injections

Les injections sont la famille de vulnérabilités la plus critique. Un attaquant fait interpréter des données comme du code par votre application ou sa base de données.

Injection SQL (SQLi)

Le scan injecte des payloads SQL dans les paramètres GET, POST et les en-têtes (User-Agent, Referer, X-Forwarded-For). Il détecte :

  • Erreurs de base de données dans la réponse (messages MySQL, PostgreSQL, MSSQL, OracleDB, SQLite)
  • Injections time-based : si un payload SLEEP(1) / pg_sleep(1) provoque un délai de réponse anormal, la vulnérabilité est confirmée

Sévérité : Critique. Une SQLi permet de lire, modifier ou supprimer toute la base de données, voire d'exécuter des commandes système.

Injection NoSQL (NoSQLi)

Pour les APIs JSON, le scan teste des opérateurs MongoDB ($gt, $where, $regex) dans les paramètres et corps de requête pour contourner les filtres de requête.

Sévérité : Élevée à critique selon le contournement possible.

Injection de commande OS

Des séparateurs de commande (;, &&, |, backticks) combinés à des commandes Unix (id, sleep, whoami) sont injectés dans les paramètres. La détection se base sur les réponses et les délais (time-based pour les commandes non réfléchies).

Sévérité : Critique. Exécution de code arbitraire sur le serveur.

Server-Side Template Injection (SSTI)

Des expressions de template ({{7*7}}, #{7*7}, ${7*7}) sont injectées dans les paramètres. Si la réponse contient 49 ou une erreur de template, le moteur de rendu est vulnérable (Jinja2, Twig, FreeMarker, etc.).

Sévérité : Critique. SSTI mène directement à l'exécution de code côté serveur.

XXE (XML External Entity)

Pour les endpoints qui acceptent du XML, le scan envoie une entité externe (&xxe;) définie dans le DOCTYPE. Si la réponse contient le contenu de l'entité ou produit une erreur caractéristique, le parseur XML est vulnérable.

Sévérité : Critique. Permet de lire des fichiers locaux du serveur, de déclencher des SSRF ou de provoquer un déni de service (billion laughs).

LFI / RFI (Local / Remote File Inclusion)

Des séquences de traversée de répertoire et des chemins de fichiers système (/etc/passwd, C:\Windows\win.ini) sont testés dans les paramètres qui semblement référencer des fichiers. La détection cherche des signatures de fichiers système dans les réponses.

Sévérité : Critique (LFI peut lire des secrets serveur) à élevée (RFI peut inclure du code distant).

Path Traversal

Des séquences ../ encodées de multiples façons sont injectées dans les paramètres de chemin pour tenter d'accéder à des fichiers en dehors du répertoire web.

Sévérité : Élevée. Lecture de fichiers de configuration, logs ou secrets hors du répertoire autorisé.

Ce qui peut fausser les résultats

Un WAF actif peut bloquer les payloads avant qu'ils n'atteignent votre application — les injections n'apparaîtront pas dans les résultats même si le code est vulnérable. Un WAF qui masque les erreurs SQL sans corriger le code sous-jacent peut aussi masquer une vulnérabilité réelle.

6. Contrôle d'accès

CORS actif (réflexion + credentials)

Le scan envoie des requêtes avec un en-tête Origin contrôlé vers votre application et vérifie la réponse :

  • Réflexion d'origine + Credentials : si l'application répond avec Access-Control-Allow-Origin: [votre-origin] et Access-Control-Allow-Credentials: true, un site malveillant peut effectuer des requêtes authentifiées au nom de vos utilisateurs et lire les réponses.
  • Wildcard sur endpoint sensible : Access-Control-Allow-Origin: * sur une route API expose les données de la réponse à n'importe quel site.

Sévérité : Élevée à critique selon les données exposées.

CSRF (Cross-Site Request Forgery)

Le scan analyse les formulaires POST sur les endpoints sensibles (login, changement de mot de passe, paiement) à la recherche :

  • De l'absence de token CSRF ou d'en-tête anti-CSRF
  • De l'absence de l'attribut SameSite sur les cookies de session (qui offre une protection partielle)

Un endpoint vulnérable au CSRF peut être déclenché à l'insu de l'utilisateur depuis n'importe quel site.

Sévérité : Moyenne à élevée selon les actions possibles.

SSRF (Server-Side Request Forgery)

Des URLs vers des adresses internes (http://169.254.169.254/ — AWS metadata, http://localhost/, http://192.168.x.x/) sont injectées dans les paramètres qui semblent contenir des URLs. Si la réponse révèle un contenu interne ou une erreur de connexion caractéristique, le check est positif.

Sévérité : Critique dans les environnements cloud. SSRF permet d'atteindre les services internes, les métadonnées cloud (clés IAM, tokens) et les services non exposés publiquement.

Host Header Injection

Le scan envoie des requêtes avec un en-tête Host falsifié. Si la réponse réfléchit la valeur du Host ou génère des URLs basées dessus, l'application peut être vulnérable à des attaques de type password-reset poisoning (envoi de liens de réinitialisation pointant vers un domaine attaquant).

Sévérité : Moyenne à élevée selon l'usage.

Redirections ouvertes (Open Redirect)

Les paramètres de redirection (redirect, next, url, return, etc.) sont testés avec des URLs externes contrôlées. Si l'application redirige vers une URL tierce sans validation, un attaquant peut créer des liens de phishing sur votre domaine de confiance.

Sévérité : Faible à moyenne. Facilite les attaques de phishing en abusant de la confiance dans votre domaine.

Méthodes HTTP dangereuses

Le scan envoie des requêtes OPTIONS puis teste PUT, DELETE, TRACE sur les endpoints détectés. TRACE peut être exploité pour voler des cookies via une injection de script (attaque XST). PUT/DELETE non protégés exposent des ressources à la modification ou suppression.

Sévérité : Faible à moyenne. À désactiver si non nécessaire.

7. Authentification & Sessions

IDOR / BOLA / BFLA

Le scan tente d'accéder à des ressources appartenant à d'autres utilisateurs en modifiant des identifiants numériques (/api/users/1, /api/orders/42) dans les URLs. Il vérifie également si des endpoints d'administration sont accessibles sans élévation de privilège.

  • IDOR (Insecure Direct Object Reference) : accès à un objet par son ID sans vérification des droits
  • BOLA (Broken Object Level Authorization) : variante API-first de l'IDOR
  • BFLA (Broken Function Level Authorization) : accès à des fonctions d'admin sans les droits requis

Sévérité : Élevée à critique. Ces failles représentent la première catégorie de vulnérabilité dans l'OWASP API Security Top 10.

Bruteforce / Lockout

Le scan envoie plusieurs tentatives de connexion successives avec des identifiants invalides et vérifie si l'application bloque ou ralentit les tentatives après N échecs. L'absence de mécanisme de verrouillage de compte expose l'application aux attaques par dictionnaire.

Sévérité : Élevée. Sans lockout, un attaquant peut tester des milliers de combinaisons mot de passe/login.

Gestion des sessions / JWT

Le scan vérifie :

  • Non-invalidation de session : après un appel de déconnexion, l'ancien cookie de session est-il encore accepté ?
  • Expiration JWT absente : des tokens JWT sans champ exp restent valides indéfiniment
  • Algorithme JWT none : acceptation de tokens non signés (vulnérabilité connue des librairies JWT mal configurées)
  • Fixation de session : l'identifiant de session change-t-il après authentification ?

Sévérité : Élevée. Un token non invalidé permet à un attaquant de continuer à utiliser une session volée même après que la victime s'est déconnectée.

OAuth / OIDC

Le scan teste les flux OAuth courants à la recherche de :

  • Absence de vérification du paramètre state (protection CSRF du flux OAuth)
  • Redirection vers des URIs non enregistrées (redirect_uri non validée)
  • Tokens d'accès exposés dans les URLs (dans les logs, l'historique du navigateur)

Sévérité : Élevée. Ces failles peuvent permettre le vol de tokens d'accès OAuth et la compromission de comptes.

8. Injections côté client (XSS réfléchi)

Le scan injecte des marqueurs XSS (<script>alert(1)</script>, des variantes encodées) dans les paramètres GET et POST. Il analyse ensuite la réponse HTML pour détecter si le marqueur est réfléchi sans encodage.

Ce qui est testé

  • Paramètres de recherche (?q=, ?search=, ?keyword=)
  • Paramètres de navigation (?page=, ?sort=, ?filter=)
  • Corps de formulaires HTML

Pourquoi le XSS réfléchi est dangereux

Un XSS réfléchi permet à un attaquant d'envoyer un lien piégé à une victime. Quand la victime clique, le script s'exécute dans son navigateur avec les droits de votre site : vol de cookies de session, redirection vers un site de phishing, exécution d'actions à son nom.

Sévérité : Élevée. Si des cookies de session sans HttpOnly sont présents, la sévérité monte à critique.

Limites du check XSS

Le scan détecte la réflexion directe dans les réponses HTML. Il ne détecte pas le XSS stocké (injecté en base de données) ni les DOM-based XSS (manipulation côté JavaScript). Ces variantes nécessitent une analyse plus poussée avec un navigateur complet.

9. Logique métier & Données

Logique métier (Business Logic)

Le scan sonde les endpoints de type transaction pour détecter :

  • Montants/quantités négatifs acceptés : envoyer quantity=-1 ou amount=-100 dans une commande peut créer un crédit non prévu
  • Manipulation d'état de workflow : accès direct à une étape de processus métier en ignorant les étapes précédentes

Sévérité : Élevée à critique. Ces failles sont spécifiques à votre application et souvent invisibles aux outils automatiques classiques.

Race Conditions

Le scan envoie des requêtes en parallèle vers des endpoints critiques (validation de coupon, achat, transfert) pour détecter des comportements incohérents dus à des accès concurrents non synchronisés. Une race condition peut permettre de dépenser le même coupon deux fois ou de retirer plus que son solde.

Sévérité : Élevée selon les fonctionnalités concernées.

DoS / Rate Limiting

Le scan envoie un burst de 20 requêtes rapides vers l'URL cible pour détecter l'absence de protection contre les abus de taux. L'absence de réponse 429 Too Many Requests après ce burst indique qu'un attaquant peut saturer l'application ou contourner les limites d'usage.

Sévérité : Faible à moyenne. Un endpoint non rate-limité peut être abusé pour du scraping, de l'énumération ou du DoS.

Upload abusif

Le scan teste les endpoints d'upload de fichiers avec :

  • Des fichiers dont l'extension et le Content-Type sont incohérents (ex. un fichier PHP déguisé en image)
  • Des fichiers volumineux pour tester les limites de taille
  • Des chemins avec traversée de répertoire dans le nom de fichier

Sévérité : Critique si un fichier PHP/JSP peut être uploadé et exécuté — cela mène à l'exécution de code arbitraire.

Désérialisation non sécurisée

Le scan détecte les endpoints qui acceptent des objets sérialisés (Java, PHP, Python pickle, YAML) et envoie des payloads malformés connus pour déclencher des erreurs caractéristiques. La détection reste conservative (faux positifs rares) — elle ne tente pas d'exploitation réelle.

Sévérité : Critique. La désérialisation non sécurisée est l'une des voies d'accès les plus directes à l'exécution de code côté serveur.

Mass Assignment

Le scan ajoute des champs supplémentaires dans les requêtes PUT/PATCH (role, is_admin, price, balance) pour détecter si l'application les accepte sans filtrage. Un framework qui bind directement le corps JSON aux propriétés d'un objet est vulnérable si les champs non attendus ne sont pas explicitement ignorés.

Sévérité : Élevée. Permet d'escalader ses privilèges ou de manipuler des données sensibles.

10. APIs & Protocoles modernes

Ces checks ciblent les endpoints API et protocoles avancés. Ils sont activés pour les cibles Backend.

GraphQL — Introspection exposée

L'introspection GraphQL permet de lister tous les types, champs et mutations de votre schéma. Le scan détecte si elle est activée en production en envoyant une requête d'introspection standard. Un schéma exposé est un guide d'attaque complet pour votre API.

Sévérité : Moyenne. À désactiver en production.

GraphQL — Abus de profondeur

Le scan envoie des requêtes GraphQL profondément imbriquées pour tester si l'application limite la profondeur des requêtes. Sans limite, une requête avec 50 niveaux d'imbrication peut provoquer une charge exponentielle (DoS).

Sévérité : Moyenne.

GraphQL — Souscriptions sans authentification

Le scan tente de s'abonner à des souscriptions GraphQL sans fournir de credentials. Si des données en temps réel sont accessibles anonymement, cela peut exposer des événements sensibles.

Sévérité : Élevée si des données utilisateur sont exposées.

gRPC Reflection exposée

Le service gRPC Reflection liste tous les services et méthodes disponibles sur votre serveur gRPC. Le scan détecte si ce service est actif en production. Comme l'introspection GraphQL, c'est un guide d'attaque complet pour votre API gRPC.

Sévérité : Moyenne. À désactiver en production.

WebSocket — Autorisation

Le scan teste si votre endpoint WebSocket accepte des connexions sans vérification de l'en-tête Origin ou sans authentification, ce qui permettrait à un site malveillant d'établir une connexion WebSocket au nom d'un utilisateur authentifié (WebSocket hijacking).

Sévérité : Élevée selon les données échangées sur le canal.

Schéma OpenAPI sans validation

Le scan récupère votre schéma OpenAPI exposé et teste si des endpoints documentés acceptent des données ne respectant pas le schéma (types, formats, contraintes). Une validation insuffisante peut permettre des injections ou des comportements non prévus.

Sévérité : Variable. Le schéma OpenAPI accessible publiquement est lui-même un finding informatif.

11. Infrastructure & Cache

Cache Poisoning

Le scan injecte des en-têtes de proxy (X-Forwarded-Host, X-Host, X-Forwarded-Scheme) avec des valeurs contrôlées et vérifie si la réponse les réfléchit. Si le cache mémorise une réponse empoisonnée, tous les utilisateurs qui accèdent ensuite à la même page reçoivent la version malveillante.

Sévérité : Élevée. Impact en cascade sur tous les utilisateurs servis par le cache.

Web Cache Deception

Le scan accède à des URLs comme /profil/faux.css pour détecter si votre application sert des pages dynamiques authentifiées sous des extensions statiques (CSS, JS, PNG) qui seraient mises en cache publiquement. Cela permettrait à un attaquant d'accéder à des pages privées servies depuis le cache.

Sévérité : Élevée. Exposition de données utilisateur via un cache public.

Request Smuggling (CL/TE et TE/TE)

Le scan envoie des requêtes avec des en-têtes Content-Length et Transfer-Encoding ambigus pour tester si le proxy front-end et le serveur back-end désaccordent la limite des requêtes. Un désaccord permet d'injecter des données dans la requête d'un autre utilisateur.

Sévérité : Critique. Permet de contourner des protections de sécurité, d'injecter des requêtes malveillantes ou de voler des sessions d'autres utilisateurs.

Endpoints internes exposés

Le scan sonde des chemins connus pour exposer des informations de diagnostic en production : /actuator, /actuator/env, /actuator/health, /_debug, /debug/vars, /metrics, /__admin. Ces endpoints Spring Boot, Django Debug Toolbar ou autres frameworks révèlent souvent des variables d'environnement, des connexions internes ou des métriques internes.

Sévérité : Élevée. /actuator/env peut contenir des mots de passe de base de données, des clés API, des secrets de configuration.

Object Storage public (S3 / GCS / Azure Blob)

Le scan détecte les buckets de stockage référencés dans les réponses et tente de lister leur contenu. Un bucket public en listing expose tous les fichiers qu'il contient — documents utilisateur, exports, backups.

Sévérité : Critique si le listing est actif. Élevée si le bucket existe mais est correctement protégé.

12. Comprendre votre score intrusif

Niveaux de sévérité

Identiques au scan passif, les niveaux de sévérité reflètent la gravité d'une vulnérabilité exploitable :

Sévérité Pénalité Description
Critique100 pointsExploitation directe : exécution de code, accès total à la base de données, prise de contrôle de compte. À corriger immédiatement.
Élevée (high)50 pointsAccès non autorisé à des données, contournement d'authentification, exposition grave. Traitement prioritaire.
Moyenne (medium)25 pointsVulnérabilité exploitable dans des conditions spécifiques. À planifier rapidement.
Faible (low)10 pointsSurface d'attaque élargie sans exploitation directe évidente. Bonne pratique à corriger.
Informatif (info)0 pointObservation de sécurité, configuration révélatrice mais non exploitable directement.

Pondération par macro-catégorie

Les catégories les plus critiques (injections, SSRF, IDOR) ont un poids plus élevé dans le calcul du score global :

CatégorieChecksPoids relatif
Injections (SQLi, NoSQLi, commandes, SSTI, XXE, LFI, Path Traversal)14 checksÉlevé
Authentification & Sessions (IDOR, Bruteforce, JWT, OAuth)7 checksÉlevé
Contrôle d'accès (CORS, CSRF, SSRF, Host Header, Redirections, Méthodes HTTP)6 checksÉlevé
Logique métier & Données (Race, DoS, Upload, Désérialisation, Mass Assignment)7 checksMoyen
Injections côté client (XSS réfléchi)1 checkMoyen
Infrastructure & Cache (Cache Poisoning, Smuggling, Endpoints internes, Storage)5 checksMoyen
APIs & Protocoles modernes (GraphQL, gRPC, WebSocket, OpenAPI)6 checksFaible à moyen

Interpréter le score

Un score de 100 indique qu'aucune anomalie n'a été détectée par les checks du scan intrusif. C'est un bon signal, pas une garantie absolue : le scan couvre les vulnérabilités détectables automatiquement avec des requêtes in-band. Des failles complexes de logique métier ou des vulnérabilités out-of-band peuvent nécessiter un audit manuel complémentaire.

En priorité : corrigez toujours les findings critiques en premier, quel que soit leur impact sur le score. Une injection SQL critique est plus urgente qu'une dizaine de findings faibles.

13. Limites et faux positifs

Ce que le scan ne peut pas détecter

  • Vulnérabilités out-of-band : les payloads qui exfiltrent des données via des callbacks DNS ou HTTP externe ne sont pas détectés (prévu dans une version ultérieure)
  • XSS stocké : nécessite un navigateur pour observer l'exécution dans le DOM
  • Failles dans le code non accessibles par HTTP : vulnérabilités dans des jobs asynchrones, workers, processus non exposés
  • Logique métier complexe : le scan peut détecter des patterns courants mais ne comprend pas la sémantique spécifique à votre domaine
  • Authentification multi-facteurs : les flux MFA ne peuvent pas être simulés automatiquement

Gestion des faux positifs

Certains findings peuvent être des faux positifs selon votre contexte :

  • Rate limiting absent : si vous avez une protection rate limiting côté CDN ou load balancer (Cloudflare, Nginx), elle peut ne pas être visible au niveau de l'application et générer un false positive
  • Méthodes HTTP dangereuses : PUT/DELETE sont normaux et attendus sur une API REST bien conçue — leur présence n'est un problème que si elle n'est pas protégée par authentification
  • IDOR / BFLA : les checks testent l'accès à des IDs séquentiels courants. Sur un système avec des IDs UUID ou des tokens opaques, les checks peuvent ne pas couvrir tous les cas

14. Ce qui peut bloquer ou impacter le scan

WAF et protections anti-bot

Un Web Application Firewall (Cloudflare, AWS WAF, ModSecurity) peut bloquer les payloads d'injection avant qu'ils n'atteignent votre application. Le scan recevra des réponses 403 ou des pages de challenge à la place des réponses applicatives. Dans ce cas :

  • Les injections n'apparaîtront pas dans les résultats même si le code est vulnérable
  • C'est un signe que votre WAF fonctionne — mais le code sous-jacent reste à corriger
  • Pour un test complet, configurez une liste d'IP autorisées pour bypasser le WAF pendant le scan

Timeouts

Le scan intrusif a un budget de temps plus élevé que le scan passif (jusqu'à 8 minutes). Les checks time-based (SQLi blind, injection de commande) attendent intentionnellement des délais anormaux. Sur une application très lente, des faux positifs de type time-based peuvent apparaître.

Environnements sans paramètres

Certains checks (injections, XSS réfléchi) nécessitent des paramètres dans l'URL ou des formulaires HTML pour avoir quelque chose à tester. Sur une page statique sans paramètres, ces checks seront exécutés mais ne produiront aucun résultat (ni positif ni négatif).

APIs REST-only sans documentation

Sur une API sans Swagger/OpenAPI exposé, le scanner ne peut explorer que l'URL fournie et les chemins qu'il tente de déduire. Pour une couverture complète d'une API, utilisez la fonctionnalité d'import de documentation API (OpenAPI/Postman) pour fournir la liste des endpoints à scanner.

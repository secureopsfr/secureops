Scan passif

Le scan passif analyse la posture de sécurité de votre site web en observant ce qu'il expose : chiffrement, configuration, fichiers accessibles, en-têtes HTTP, cookies… Aucune attaque n'est lancée, aucune donnée n'est modifiée. Le scanner écoute et mesure.

1. Principe du scan passif

Le scan passif est une analyse en lecture seule. Il envoie des requêtes HTTP ordinaires — comme le ferait un navigateur ou un moteur de recherche — et examine les réponses : codes de statut, en-têtes, contenu HTML, cookies, certificats. Il ne tente pas d'injection, ne modifie aucune donnée, et ne se connecte jamais à un compte.

C'est le point de départ recommandé pour toute évaluation de sécurité. Il cartographie ce que votre site révèle de lui-même à n'importe quel visiteur, sans avoir à exploiter quoi que ce soit.

Ce que le scan passif détecte

  • Les mauvaises configurations (HTTPS absent, certificat expiré, en-têtes manquants)
  • Les fichiers ou répertoires accessibles par erreur
  • Les informations que votre serveur révèle sur lui-même (technologies, versions)
  • Les politiques trop permissives (CORS ouvert, cookies mal protégés)
  • Les ressources chargées de manière non sécurisée

Ce que le scan passif ne fait pas

  • Il ne teste pas les injections (SQL, XSS, etc.) — c'est le rôle du scan intrusif
  • Il ne se connecte pas à votre application
  • Il ne modifie, ne supprime et ne crée aucune donnée
  • Il ne peut pas voir les pages protégées par authentification

2. Scan frontend et scan backend : quelle différence ?

Un site web est composé de deux parties :

  • Le frontend : ce que l'utilisateur voit — les pages HTML, les styles, les scripts, la page d'accueil, les formulaires. C'est ce qui est servi directement au navigateur, souvent sur votresite.com.
  • Le backend : les services qui font tourner l'application — l'API qui répond aux formulaires, la base de données, les endpoints qui renvoient les données. Souvent sur api.votresite.com ou accessible via des routes /api/.

Le scan frontend passif analyse la partie visible de votre site : HTTPS, en-têtes de sécurité, cookies, ressources chargées par les pages, fichiers exposés. C'est ce que voit tout visiteur dès qu'il charge votre site.

Le scan backend passif ciblera vos APIs et services. Il est prévu dans la roadmap SecureOps. En attendant, si votre API est accessible depuis la même URL (ex. /api/), certaines vérifications — CORS, méthodes HTTP, exposition de données — couvrent déjà ces endpoints.

Conseil pratique

Commencez toujours par le scan frontend. La grande majorité des risques visibles depuis internet — certificat, en-têtes, fichiers exposés, cookies — sont détectés dès ce premier niveau. Le score obtenu est un indicateur fiable de votre exposition publique.

3. Scanner une page ou plusieurs pages

Vous avez deux façons de lancer un scan :

Mode « page unique »

Cochez « Scanner uniquement cette page ». Le scan s'exécute directement sur l'URL que vous avez saisie. C'est le mode le plus rapide, idéal pour tester une page précise (formulaire de connexion, page d'accueil, endpoint API).

Mode « multi-pages » (avec découverte)

Laissez la case décochée. Avant de lancer le scan, une phase de découverte explore votre site pour trouver les pages accessibles. Vous obtenez une liste d'URLs que vous pouvez ajuster, puis vous validez et lancez le scan.

Ce mode est utile quand vous voulez une vision globale du site : la homepage, les pages produits, le blog, les formulaires — chaque page peut avoir sa propre configuration (cookies, en-têtes, ressources) et mérite d'être analysée.

Situation Mode recommandé
Tester une page précise ou une URL d'API Page unique
Obtenir une vue d'ensemble du site Multi-pages avec découverte
Site en React / Next.js / Vue (liens générés par JavaScript) Multi-pages, mode SPA activé
Site avec peu de pages ou URLs connues Page unique, répétez sur chaque URL importante

À noter sur le scan multi-pages

Actuellement, le scan analyse chaque page de la liste de façon indépendante. Le score final reflète l'URL de départ. Les résultats des pages supplémentaires sont présentés séparément. Cette vue consolidée sera enrichie dans les prochaines versions.

La découverte respecte votre site

La phase de découverte lit le fichier robots.txt de votre site et ne suit pas les chemins marqués Disallow. Elle ne dépasse pas 2 niveaux de profondeur par défaut et reste limitée à votre domaine. Si une page importante n'apparaît pas dans la liste, vous pouvez l'ajouter manuellement avant de lancer le scan.

4. TLS et HTTPS

Ce qui est vérifié

  • HTTPS activé : votre site doit être accessible en HTTPS (chiffrement de la connexion).
  • Redirection HTTP → HTTPS : si un visiteur tape http:// ou clique sur un lien non sécurisé, il doit être redirigé vers HTTPS automatiquement.
  • Certificat valide : non expiré, non auto-signé, avec une chaîne de certification complète.
  • Alerte expiration proche : si votre certificat expire dans moins de 30 jours, une alerte est remontée.
  • Versions TLS obsolètes désactivées : TLS 1.0 et 1.1 sont dépréciés et ne doivent plus être acceptés par votre serveur.

Pourquoi c'est important

Sans HTTPS, toutes les données échangées entre votre site et vos visiteurs circulent en clair : mots de passe, données de formulaire, cookies de session. Un attaquant connecté au même réseau peut tout intercepter. Un certificat invalide affiche des avertissements de sécurité dans le navigateur et dégrade la confiance de vos utilisateurs.

Ce qui peut bloquer le scan

  • Site accessible uniquement en HTTP : le scan peut s'exécuter mais signalera un risque critique. Passez en HTTPS en priorité.
  • Certificat expiré ou auto-signé : le scan continue mais remonte des anomalies. Vos utilisateurs voient probablement des avertissements dans leur navigateur.

Conseils

Utilisez un certificat délivré par une autorité reconnue (Let's Encrypt est gratuit). Configurez votre serveur pour rediriger tout le port 80 vers le port 443. Désactivez TLS 1.0 et 1.1 dans la configuration SSL de votre serveur (Nginx, Apache, Caddy).

5. En-têtes de sécurité

Ce qui est vérifié

  • Content-Security-Policy (CSP) : définit les sources autorisées pour les scripts, styles, images. Réduit considérablement l'impact des attaques XSS.
  • Strict-Transport-Security (HSTS) : force le navigateur à n'accéder au site qu'en HTTPS pendant une durée définie. Protège contre les attaques de dégradation vers HTTP.
  • X-Frame-Options : empêche votre page d'être intégrée dans une iframe sur un autre site (protection contre le clickjacking, où un attaquant superpose une interface invisible).
  • X-Content-Type-Options : empêche le navigateur de « deviner » le type d'un fichier. Sans cet en-tête, un fichier texte uploadé pourrait être interprété comme un script.
  • Referrer-Policy : contrôle les informations envoyées dans l'en-tête Referer lors de la navigation. Protège contre les fuites d'URLs sensibles vers des sites tiers.
  • Permissions-Policy : restreint l'accès aux fonctionnalités sensibles du navigateur (caméra, micro, géolocalisation, etc.) même pour des scripts tiers éventuellement compromis.

Pourquoi c'est important

Ces en-têtes sont des couches de défense supplémentaires. Ils ne remplacent pas une application bien écrite, mais ils limitent les dégâts en cas de faille : une CSP stricte rend une attaque XSS inefficace même si une vulnérabilité existe. HSTS empêche un « SSL stripping » qui forcerait une connexion non chiffrée.

Ce qui peut bloquer le scan

L'absence de ces en-têtes ne bloque pas le scan. Mais chaque en-tête manquant génère un finding. Les plus importants (CSP, HSTS, X-Frame-Options) ont une sévérité modérée à élevée.

Conseils

Ajoutez ces en-têtes via la configuration de votre serveur ou de votre framework. Commencez par X-Content-Type-Options: nosniff et X-Frame-Options: DENY (simples à ajouter). Passez ensuite à HSTS, puis à la CSP, qui demande plus de travail mais apporte le plus de valeur.

6. Cookies

Ce qui est vérifié

  • Flag Secure : les cookies doivent être marqués Secure pour n'être transmis qu'en HTTPS. Sans ce flag, un cookie peut circuler en clair.
  • Flag HttpOnly : empêche JavaScript d'accéder au cookie. Protège contre le vol de session en cas de script malveillant injecté dans la page.
  • Attribut SameSite : contrôle l'envoi du cookie lors de requêtes venant d'un autre site. La valeur Strict ou Lax protège contre les attaques CSRF (requêtes forgées).
  • Triple protection de session : les cookies de session doivent cumuler HttpOnly + Secure + SameSite=Strict. Le scan alerte si l'un de ces trois flags est absent.
  • Préfixes sécurisés : les cookies sensibles peuvent utiliser les préfixes __Host- ou __Secure- pour renforcer les garanties de transmission.
  • Durée de session : un cookie de session avec une date d'expiration supérieure à 24h est signalé. Les sessions longues augmentent la fenêtre de risque en cas de vol.
  • Cookies tiers (CHIPS) : les cookies tiers sans l'attribut Partitioned sont signalés conformément aux nouvelles exigences des navigateurs.

Pourquoi c'est important

Un cookie de session volé permet à un attaquant de se connecter à votre application en se faisant passer pour l'un de vos utilisateurs. Sans HttpOnly, un script injecté suffit à le voler. Sans Secure, il peut être intercepté sur une connexion HTTP. Sans SameSite, un lien piégé sur un autre site peut déclencher des actions à l'insu de votre utilisateur connecté.

Conseils

Pour tous vos cookies de session et d'authentification, appliquez systématiquement Secure; HttpOnly; SameSite=Strict. Pour les cookies dont vous avez besoin en cross-site (par exemple un cookie de suivi de panier partagé), utilisez SameSite=Lax comme compromis.

7. Fichiers sensibles exposés

Ce qui est vérifié

Le scan teste une liste de chemins connus pour être exposés accidentellement :

  • /.env — fichier de configuration contenant mots de passe, clés API, connexions base de données
  • /.git/config — dépôt Git exposé (accès à l'historique du code source)
  • /backup.zip, /backup.sql, /config.bak, /.env.bak — sauvegardes
  • /phpinfo.php — page de diagnostic PHP révélant la configuration serveur
  • /admin/ — interface d'administration accessible publiquement
  • /.htaccess, /web.config — fichiers de configuration serveur
  • /composer.json, /package.json, /.npmrc — dépendances du projet exposées
  • /.DS_Store — métadonnées macOS révélant la structure de fichiers
  • /.svn/entries — dépôt SVN exposé

Un fichier est considéré exposé uniquement si la réponse est HTTP 200 et que le contenu correspond aux signatures attendues. Le scan évite ainsi les faux positifs liés à des pages 404 personnalisées.

Pourquoi c'est important

Ces fichiers ne devraient jamais être accessibles depuis internet. Un fichier .env expose directement vos secrets (identifiants de base de données, clés d'API tierces, clés de chiffrement). Un répertoire .git exposé peut révéler l'intégralité de votre code source, y compris l'historique des commits. Une interface /admin/ accessible sans authentification est une porte d'entrée directe pour un attaquant.

Ces findings sont prioritaires

Toute exposition détectée est classée en sévérité critique ou élevée. Si vous obtenez ce type de finding, agissez immédiatement : supprimez les fichiers du répertoire web ou configurez votre serveur pour bloquer l'accès (réponse 403 ou 404).

Conseils

Ne placez jamais .env, .git, les backups ou phpinfo.php dans un dossier servi par votre serveur web. Dans Nginx, ajoutez des règles deny all sur ces chemins. Dans Apache, utilisez des directives Require all denied dans .htaccess.

8. Listage de répertoires

Ce qui est vérifié

Le scan teste si des répertoires sensibles affichent leur contenu (liste des fichiers) quand on y accède directement :

  • /uploads/, /assets/, /static/ — ressources publiques
  • /tmp/, /logs/, /config/, /backup/, /data/ — répertoires techniques

Le scan signale également un code 403 sur les répertoires sensibles (/config/, /backup/, /logs/, etc.) : même si l'accès est bloqué, le 403 confirme l'existence du répertoire à un attaquant.

Pourquoi c'est important

Avec le listage activé sur /uploads/, n'importe qui peut voir la liste de tous les fichiers uploadés par vos utilisateurs, y compris des documents confidentiels. Sur /logs/, un attaquant peut lire vos journaux d'application et y trouver des informations d'erreur, des traces de requêtes ou des données personnelles.

Conseils

Désactivez le directory listing au niveau du serveur (option Indexes dans Apache, autoindex off dans Nginx). Placez un fichier index.html vide dans les répertoires qui doivent être accessibles mais sans afficher leur contenu.

9. robots.txt

Ce qui est vérifié

  • Présence et lecture : le scan récupère le fichier /robots.txt et extrait les directives Disallow, Allow, Sitemap et Crawl-delay.
  • Chemins sensibles dans Disallow : si robots.txt mentionne des chemins comme /admin/, /api/, /config/, /backup/, ces chemins sont signalés. Non pas parce qu'ils sont menacés, mais parce que leur existence devient publique.

Pourquoi c'est important

robots.txt n'est pas une mesure de sécurité. Il indique aux moteurs de recherche quelles pages ne pas indexer, mais n'importe qui peut lire ce fichier et ignorer ses directives. Mentionner /admin/ ou /api/v2/ dans Disallow revient à publier un plan de votre site aux attaquants. Ces chemins doivent être protégés par authentification, pas seulement exclus de l'indexation.

Sévérité

Les findings liés à robots.txt sont de niveau informatif ou faible. L'absence de robots.txt n'est pas un problème de sécurité.

10. Sitemap

Ce qui est vérifié

  • Localisation : le scan cherche le sitemap via robots.txt (directive Sitemap:), puis en fallback à l'emplacement classique /sitemap.xml.
  • URLs sensibles dans le sitemap : si le sitemap référence des chemins sensibles (/admin, /api, /config, etc.), le scan les signale. Ces URLs seraient indexées par les moteurs de recherche et visibles publiquement.

Pourquoi c'est important

Un sitemap sert à aider Google et les autres moteurs de recherche à découvrir vos pages. Si une page d'administration ou un endpoint API se retrouve dans le sitemap, elle apparaîtra dans les résultats de recherche et sera facilement trouvée. Votre sitemap devrait ne contenir que les pages destinées au grand public.

Conseils

Ne listez dans votre sitemap que les pages publiques. Excluez /admin/, /api/, les pages de confirmation ou de panier. La plupart des frameworks (WordPress, Next.js, etc.) ont des options pour contrôler ce qui est inclus dans le sitemap.

11. CORS et cross-origin

Ce qui est vérifié

Le scan envoie des requêtes GET et OPTIONS avec un en-tête Origin vers votre page principale et vers des chemins sensibles dérivés (/api/, /user/, /admin/, etc.) pour tester votre configuration CORS (Cross-Origin Resource Sharing).

  • CORS ouvert sur endpoint sensible : si un de vos endpoints répond avec Access-Control-Allow-Origin: *, n'importe quel site peut lire ses réponses depuis le navigateur de vos utilisateurs.
  • Réflexion d'origine avec Credentials : si le serveur répercute l'origine de la requête (Access-Control-Allow-Origin: [origine appelante]) en autorisant les cookies (Access-Control-Allow-Credentials: true), un site malveillant peut effectuer des requêtes authentifiées au nom de vos utilisateurs et lire les données.
  • Contenu mixte : des ressources (images, scripts, styles) chargées en HTTP sur une page HTTPS affaiblissent la sécurité de la page entière. Le navigateur peut bloquer ces ressources ou afficher des avertissements.
  • Cross-Origin-Resource-Policy (CORP) : absence de cet en-tête sur des ressources sensibles, qui permettrait à des sites tiers de les intégrer.

Pourquoi c'est important

Une mauvaise configuration CORS est l'une des vulnérabilités les plus sous-estimées. Elle peut permettre à un site malveillant de lire les données de votre API au nom d'un utilisateur connecté : emails, informations de compte, historique de commandes. Le contenu mixte dégrade la protection HTTPS de l'ensemble de la page.

Conseils

Limitez CORS à une liste explicite de domaines de confiance. N'utilisez jamais Access-Control-Allow-Origin: * sur des endpoints qui retournent des données utilisateur ou qui acceptent des cookies. Pour les ressources publiques (images d'un CDN, polices), le wildcard est acceptable.

12. Méthodes HTTP et redirections

Ce qui est vérifié

  • Méthodes dangereuses exposées : si votre serveur autorise PUT, DELETE ou PATCH sans protection, un attaquant peut tenter de modifier ou supprimer des ressources.
  • Méthode TRACE activée : TRACE est une méthode HTTP de diagnostic qui peut être détournée pour voler des cookies ou des tokens d'authentification (attaque XST). Elle ne devrait jamais être activée en production.
  • Chaînes de redirection excessives : plus de 3 redirections successives pour atteindre une page ralentissent le chargement et peuvent masquer des configurations anormales.
  • Redirections 301/302 sur formulaires sensibles : une redirection non permanente (307/308) est préférable pour les formulaires POST, car une redirection 301/302 peut faire perdre les données du formulaire selon les navigateurs.

Pourquoi c'est important

Les méthodes HTTP non nécessaires agrandissent la surface d'attaque. TRACE activé est considéré comme une mauvaise pratique de sécurité depuis plus de 20 ans. Les chaînes de redirection excessives révèlent souvent des migrations incomplètes ou des configurations désorganisées.

Conseils

Désactivez TRACE dans la configuration de votre serveur. N'exposez que les méthodes HTTP utiles à chaque endpoint. Pour les formulaires d'authentification, utilisez des redirections 307 ou 308.

13. Cache et performances

Ce qui est vérifié

  • En-têtes de cache sur la page principale : présence et cohérence de Cache-Control, Pragma, ETag, Last-Modified, Vary.
  • Pages sensibles cacheables publiquement : une page de connexion, d'administration ou un endpoint API ne doit pas être mise en cache publiquement. Si un proxy ou un CDN cache une telle page, les données d'un utilisateur pourraient être servies à un autre.
  • Cache des assets statiques : le scan vérifie que les fichiers JavaScript, CSS et images avec un hash dans leur URL (ex. main.a3c9f7.js) bénéficient d'un cache long (max-age élevé et immutable). Sans ça, chaque visite retélécharge inutilement ces fichiers.

Pourquoi c'est important

Un cache mal configuré a deux risques opposés : des pages sensibles trop cacheables (fuite de données entre utilisateurs) et des assets statiques pas assez cacheables (performances dégradées, surcharge serveur). La sécurité et les performances vont ici de pair.

Conseils

Appliquez Cache-Control: no-store sur les pages de connexion, les pages de compte et les réponses API. Pour les assets avec hash dans l'URL, utilisez Cache-Control: max-age=31536000, immutable.

14. Intégrité des ressources

Ce qui est vérifié

  • Subresource Integrity (SRI) : les scripts et feuilles de style chargés depuis des CDN externes devraient avoir un attribut integrity contenant un hash. Si le CDN est compromis et sert un fichier modifié, le navigateur refusera de l'exécuter.
  • Scripts inline et CSP : si une Content-Security-Policy est définie, les scripts inline doivent utiliser des nonces ou des hashes pour être autorisés. Sans ça, la CSP est contournable.
  • Champs mot de passe et autocomplete : les champs de saisie de mot de passe sur des pages sensibles devraient avoir des attributs autocomplete appropriés pour éviter que le navigateur ne propose d'enregistrer ou de remplir automatiquement des identifiants sur des pages non prévues à cet effet.
  • Liens target="_blank" et noopener : un lien qui ouvre un onglet externe sans rel="noopener" permet à la page cible d'accéder à votre page via window.opener — une faille connue.
  • Meta robots sur pages sensibles : les pages de login, d'administration et les endpoints API devraient avoir une balise meta name="robots" content="noindex, nofollow" pour ne pas être indexées par les moteurs de recherche.

Pourquoi c'est important

SRI protège contre la compromission de fournisseurs CDN tiers (jQuery, Bootstrap, Google Fonts, etc.). Si un attaquant modifie un fichier JS sur un CDN, sans SRI votre site exécutera du code malveillant. Les liens target="_blank" non protégés permettent des attaques de type reverse tabnapping.

15. Empreinte technologique

Ce qui est vérifié

  • En-têtes révélateurs : Server (ex. Apache/2.4.51), X-Powered-By (ex. PHP/8.1), X-Generator.
  • Balises HTML : la balise meta name="generator" utilisée par WordPress, Drupal et autres CMS révèle la technologie et parfois la version.
  • Scripts et URLs caractéristiques : les noms des fichiers JS/CSS, les chemins de ressources, les URLs d'API identifient souvent la stack technique.
  • Versions potentiellement vulnérables : si une version identifiée est connue pour avoir des failles non corrigées, un finding est remonté avec le niveau de risque correspondant.

Pourquoi c'est important

Exposer vos technologies et versions facilite la reconnaissance pour un attaquant. Il peut chercher directement les CVE (failles connues) associées à votre version de PHP, WordPress ou Apache. En masquant ces informations, vous obligez l'attaquant à faire un travail supplémentaire et à prendre des risques supplémentaires.

Conseils

Supprimez ou masquez les en-têtes Server et X-Powered-By dans la configuration de votre serveur. Retirez la balise meta generator de votre CMS. Maintenez vos composants à jour : c'est la protection la plus efficace contre les vulnérabilités connues.

16. Fuites d'information

Ce qui est vérifié

  • Stack traces et messages d'erreur : si le corps de la réponse contient des traces d'erreur (ex. Exception in thread "main", PDOException), c'est que votre application fonctionne en mode debug. Ces messages révèlent la structure interne de votre code à quiconque provoque une erreur.
  • Secrets et tokens en clair : le scan analyse le contenu HTML et les en-têtes pour détecter des patterns ressemblant à des clés API, tokens JWT, ou secrets de configuration exposés involontairement.
  • En-têtes de débogage : des en-têtes comme X-Debug, X-Debug-Token, X-Runtime trahissent un mode de débogage actif en production.

Pourquoi c'est important

Une stack trace révèle les chemins absolus de vos fichiers, les noms de vos classes, la structure de votre base de données et la version de vos dépendances — autant d'informations utiles pour cibler une attaque. Un token JWT ou une clé API exposée dans le HTML d'une page peut être réutilisé immédiatement par un attaquant.

Mode debug en production

Si un finding de ce type est détecté, désactivez immédiatement le mode debug. C'est une erreur de configuration fréquente lors des déploiements (variable APP_DEBUG=true dans le .env, DEBUG=True en Django, etc.).

17. APIs et formats

Ce qui est vérifié

  • GraphQL introspection désactivée : l'introspection GraphQL permet de lister tous les types, champs et mutations disponibles dans votre API. En production, cette fonctionnalité devrait être désactivée pour éviter de livrer le plan complet de votre API aux attaquants.
  • Swagger / OpenAPI sans authentification : une interface Swagger accessible publiquement expose la documentation complète de votre API, y compris les endpoints sensibles. Elle devrait être protégée ou désactivée en production.
  • Listes REST non paginées : un endpoint REST qui retourne une liste sans pagination peut exposer l'intégralité d'une collection de données en une seule requête.
  • Content-Type correct : les réponses API doivent avoir un Content-Type cohérent avec leur contenu pour éviter les erreurs d'interprétation.
  • Compression activée : la vérification que gzip ou brotli est activé sur les réponses de la page principale et des APIs. Pas une faille de sécurité, mais un indicateur de maturité de la configuration serveur.

Pourquoi c'est important

Une documentation d'API publique (Swagger) est un guide d'attaque gratuit. L'introspection GraphQL ouverte permet à n'importe qui de cartographier votre schéma de données en quelques secondes. Ces problèmes sont fréquents sur des déploiements de staging ou de développement qui ont été accidentellement mis en production.

18. Comprendre votre score

Niveaux de sévérité

Chaque finding détecté est classé selon un niveau de gravité. Les niveaux et leur impact sur le score sont les suivants :

Sévérité Pénalité Description
Critique100 pointsExposition directe de secrets, accès non autorisé immédiatement possible (fichier .env, dépôt Git exposé, mode debug avec secrets). À corriger avant tout.
Élevée (high)50 pointsRisque majeur de compromission (vol de session, fuite de données, CORS dangereux). Traitement prioritaire.
Moyenne (medium)25 pointsConfiguration à risque qui peut être exploitée dans certaines conditions (en-têtes manquants, cookies partiellement protégés). À planifier.
Faible (low)10 pointsBonne pratique non respectée. Amélioration recommandée mais risque immédiat limité.
Informatif (info)0 pointObservation utile, pas nécessairement une faille (chemins dans robots.txt, stack technique identifiée). Ne réduit pas le score.

Formule de calcul du score

Le score global est une moyenne pondérée des scores par catégorie :

  1. Chaque catégorie (TLS, en-têtes, cookies, etc.) reçoit un score de 0 à 100 : score_catégorie = max(0, 100 − somme des pénalités des findings). Par exemple, deux findings « medium » dans une même catégorie = 100 − 25 − 25 = 50.
  2. Le score global = somme(score_cat × poids_cat) / somme(poids_cat), arrondi à l'entier, borné entre 0 et 100.
  3. Sans aucun finding, le score est 100. Un seul finding « critical » dans une catégorie ramène le score de cette catégorie à 0.

Pondération par catégorie

Un problème dans une catégorie à fort poids (TLS, en-têtes, cookies) impacte davantage le score global qu'un problème dans une catégorie à faible poids.

Le score sur 100

Un score de 100 signifie qu'aucune anomalie n'a été détectée sur les points vérifiés. Le score est un indicateur de tendance, pas un certificat absolu de sécurité — il reflète ce que le scan passif peut observer.

Priorisez toujours la correction des findings critiques et élevés avant d'optimiser le score global. Un finding critique unique est plus important qu'une dizaine de findings faibles.

Score par catégorie dans les résultats

Les résultats sont organisés par catégorie avec un détail des tests réussis, des anomalies et des conseils de correction. Vous pouvez ainsi identifier rapidement les domaines à améliorer en priorité.

19. Ce qui peut bloquer ou impacter le scan

Bloquant — le scan échoue

  • URL invalide ou domaine inexistant : vérifiez que l'URL est bien formée et que le domaine répond.
  • Site hors ligne ou timeout : si le serveur ne répond pas dans le délai imparti, le scan ne peut pas s'exécuter.
  • Protection anti-bot ou WAF : si votre hébergeur ou un service de protection (Cloudflare, Akamai, etc.) bloque les requêtes automatisées, le scan peut recevoir des réponses 403 ou des pages CAPTCHA au lieu du contenu réel. Les résultats seront partiels ou absents.

Résultats partiels

  • Contenu chargé uniquement par JavaScript : le scan analyse la réponse HTTP initiale. Du contenu injecté après le chargement de la page (rendu côté client) peut ne pas être visible. Les cookies et en-têtes HTTP, eux, sont toujours analysés correctement.
  • Pages derrière authentification : le scan ne se connecte pas à votre application. Les pages accessibles uniquement après login ne sont pas analysées.
  • Certificat auto-signé ou expiré : le scan continue mais le finding est remonté. Certaines vérifications dépendantes d'une connexion HTTPS valide peuvent être affectées.
  • Redirections en boucle : trop de redirections successives interrompent l'analyse de l'URL concernée.

Ce que le scan ne voit pas

  • Les vulnérabilités applicatives (injections, XSS) — couvertes par le scan intrusif
  • La logique métier de votre application
  • Les pages et données derrière une authentification
  • Les failles dans le code côté serveur non visibles dans les réponses HTTP

# Vérifications actives — Méthodes HTTP

Ce document décrit les vérifications actives liées aux méthodes HTTP : objectif, méthodologie, risques, exemples et recommandations.

---

## Contexte et méthodologie

### Objectif

Mesurer la surface d'attaque exposée par les méthodes HTTP et identifier les configurations risquées : verbes non nécessaires, TRACE activé (XST), incohérences HEAD.

### Principe (actif léger, MVP 1.0.0)

Le scanner envoie un nombre limité de requêtes :

- `OPTIONS` pour récupérer `Allow` et les méthodes déclarées ;
- `TRACE` pour détecter un éventuel écho ;
- `HEAD` pour valider le comportement basique côté serveur.

Le test ne tente aucune action destructive et reste borné en volume.

---

## 1. Requête OPTIONS : méthodes autorisées

### Résumé

La requête `OPTIONS` permet d'inventorier les méthodes accessibles. Une exposition de méthodes d'écriture non nécessaires (`PUT`, `PATCH`, `DELETE`) augmente la surface d'attaque.

### Explication détaillée

Le scanner observe :

- `Allow` (si présent) ;
- `Access-Control-Allow-Methods` (contexte CORS) ;
- différences entre endpoint racine et endpoints applicatifs.

Le finding est contextuel :

- **Info** si méthode potentiellement légitime (API REST documentée) ;
- **Low/Medium** si méthodes d'écriture exposées sur endpoint non prévu.

### Exemple

- **OK** : `Allow: GET, POST, OPTIONS`
- **Info/Finding** : `Allow: GET, POST, PUT, DELETE, PATCH, OPTIONS` sur un endpoint public non API

### Vulnérabilité et impact

- **Vraisemblance** : Forte.
- **Impact** : Faible à importante selon protections authz/rate limit.

## 2. TRACE activé : risque XST

### Résumé

La méthode `TRACE` est rarement nécessaire en production. Si elle renvoie un écho (`200`), le scanner signale un risque XST et une exposition inutile de métadonnées de requête.

### Explication détaillée

Le scanner envoie `TRACE` avec un marqueur contrôlé et vérifie :

- statut `200` ;
- présence du marqueur dans le corps ;
- éventuel écho de headers.

Réponses `405` ou `501` : comportement attendu.

### Exemple

- **OK** : `TRACE /` -> `405 Method Not Allowed`
- **Finding** : `TRACE /` -> `200` + écho du marqueur

### Vulnérabilité et impact

- **Vraisemblance** : Faible à moyenne.
- **Impact** : Significative.

### Conseils

- Désactiver `TRACE` côté serveur/proxy.
- Garder une politique de verbes minimale par endpoint.
- Contrôler les verbes au niveau gateway/WAF.

### Références

- [OWASP – Cross-Site Tracing](https://owasp.org/www-community/attacks/Cross_Site_Tracing)
- [MDN – HTTP Methods](https://developer.mozilla.org/en-US/docs/Web/HTTP/Methods)

## 3. Requête HEAD : support et cohérence

### Résumé

Le support `HEAD` est un contrôle de conformité HTTP utile. Son absence n'est pas une vulnérabilité critique mais peut signaler une implémentation incomplète.

### Exemple

- **OK** : `HEAD /` -> `200` avec headers cohérents avec `GET`
- **Info** : `HEAD /` -> `405`

### Conseils

- Supporter `HEAD` sur les ressources `GET` quand possible.
